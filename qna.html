Q1 What is Static Timing Analysis (STA)? EasyBasics ▼
Answer
Static Timing Analysis is a method of verifying the timing correctness of a digital circuit without requiring test vectors or dynamic simulation. It exhaustively checks every possible timing path in the design against timing constraints to ensure the circuit can reliably operate at the target frequency.

Key characteristics: (1) Static — no need for input stimuli, it analyzes structure; (2) Exhaustive — every path is checked, unlike simulation which is limited by test coverage; (3) Fast — scales to million-gate designs in reasonable runtime; (4) Determines setup and hold slack on all paths.
Short Answer
Exhaustive, vector-free method to verify digital circuit timing against setup/hold constraints on all paths simultaneously.
Follow-up Questions
  • How is STA different from dynamic timing simulation?
    STA checks all paths algebraically without stimuli — exhaustive but may include impossible paths. Dynamic simulation uses test vectors — more realistic but coverage-limited. STA is used for timing sign-off; simulation for functional verification. STA cannot catch glitches or races that depend on data patterns.
  • What are the limitations of STA?
    STA cannot detect: functional false paths (sensitization issues), non-timing functional bugs, glitches not related to path delay, issues requiring vector-dependent analysis. It also cannot verify asynchronous interfaces without explicit constraints — those must be verified separately.
Q2 What are the four types of timing paths in STA? EasyBasics ▼
Answer
1. Input port → Register data pin (in2reg): Path from design input port through combinational logic to a flip-flop's D pin. Constrained by set_input_delay.

2. Register output → Register data pin (reg2reg): The most common path — from a flip-flop's Q/CK to the next flip-flop's D. These form the core of internal timing analysis.

3. Register output → Output port (reg2out): From flip-flop Q through combinational logic to an output port. Constrained by set_output_delay.

4. Input port → Output port (in2out / combinational): Purely combinational path with no registers. These should be minimized in synchronous design for predictability.
Follow-up Questions
  • Which path type is usually most critical and why?
    reg2reg — because these form the majority of paths in a synchronous design, they determine the maximum operating frequency (fmax), and they must all close timing simultaneously. in2out can also be critical if the combinational depth is large.
  • Can a path start at a clock pin?
    Yes — in STA, the "startpoint" of a reg2reg path is the clock pin (CK) of the launch flip-flop, not its Q pin. The clock pin is where the launching edge is defined. Q is just the output of the propagation from CK.
Q3 What is setup time of a flip-flop? What happens when it is violated? EasyBasics ▼
Answer
Setup time (Tsu) is the minimum time the data input (D) must be stable and valid before the active clock edge arrives at the flip-flop's CK pin. It is needed for the internal latch circuitry to sense and prepare to capture the data reliably.

On violation: If data arrives too late (within the setup window), the flip-flop enters metastability — its output Q enters an indeterminate voltage state between 0 and 1, oscillating or settling slowly. This can propagate incorrect data downstream. The flip-flop will eventually resolve, but the time to resolve is statistically unbounded. In silicon, this causes functional failures that are intermittent and hard to debug.
Short Answer
Minimum data stable time BEFORE clock edge. Violation → metastability → functional failures.
Follow-up Questions
  • Can setup violations be fixed without changing the design frequency?
    Yes — by reducing combinational logic depth (faster cells, fewer gates), using higher drive strength cells, reducing net RC (rerouting), or by optimizing the clock tree (useful skew). The key is reducing data path delay OR increasing the available timing window.
  • How does the tool model setup time from a Liberty library?
    Liberty files characterize setup time as a 2D table indexed by data input slew and clock input slew. The worst-case Tsu from these tables is used in the timing check equation at each flip-flop.
Q4 What is hold time? Why are hold violations considered more serious than setup violations? EasyBasics ▼
Answer
Hold time (Th) is the minimum time the data input must remain stable and valid after the active clock edge. The flip-flop needs this time to fully latch and store the captured value before the input can change.

Why hold violations are more serious:
1. Cannot be fixed by reducing frequency — hold is a DC (speed-independent) constraint. Slowing the clock makes no difference; the path still violates at any frequency.
2. Must be fixed before tape-out — unlike setup which can be waived with frequency derating, hold violations will cause functional failures in silicon at any speed.
3. Fixes add delay permanently — hold fixes require inserting buffers, increasing power and area, sometimes causing congestion.
4. Late detection risk — hold violations often appear after CTS when real clock skew is known.
Short Answer
Min data stable time AFTER clock edge. Cannot be fixed by reducing frequency — requires added delay in silicon.
Follow-up Questions
  • Where are hold violations most likely to occur?
    Short paths with few/no logic gates between flip-flops (adjacent FFs, scan chains, clock dividers, paths after wide muxes). Also after useful skew insertion for setup fixes — improving one path's setup can create hold issues on adjacent paths.
  • What is the impact of post-CTS hold violations appearing suddenly?
    Pre-CTS, ideal clocks mean zero skew — hold is checked conservatively at the same edge. Post-CTS, real skew can make capture clock arrive later than launch clock (negative skew for hold), causing paths that passed pre-CTS to fail post-CTS. This is why post-CTS hold fixing is a critical step.
Q5 What is slack? Define WNS and TNS. EasyBasics ▼
Answer
Slack = Required Arrival Time − Actual Arrival Time
— Positive slack: timing is met (data arrives with margin to spare)
— Negative slack: timing violation (data arrives too late for setup, or too early for hold)
— Zero slack: exactly on the limit — passes but no margin

WNS (Worst Negative Slack): The single most negative slack across all timing paths in the entire design — or within a path group. It represents the worst violation and determines how far the design is from closure.

TNS (Total Negative Slack): The algebraic sum of all negative slacks across all endpoints. It measures the cumulative violation magnitude — a design with 1000 slightly-failing paths has a very large TNS even if WNS is small. Sign-off requires WNS = 0 and TNS = 0.
Short Answer
WNS = single worst path slack. TNS = sum of all negative slacks. Both must be ≥ 0 at sign-off.
Follow-up Questions
  • Which is more important for project status — WNS or TNS?
    Both matter. WNS tells you the worst case (how far you are from the hardest path closing). TNS tells you the workload — a large TNS means many paths to fix, which takes more ECO iterations even if WNS is small. Early in a project, focus on TNS to reduce total work; near tape-out, WNS drives the last mile.
  • What is FNS (Failing Negative Slack endpoints)?
    FNS or "failing endpoints" count is the number of timing endpoints (register D pins, output ports) that have negative slack. It's reported in report_timing_summary. Reducing FNS to zero is the goal. Useful alongside TNS to gauge closure progress.
Q6 Write the setup timing check equation from first principles. MediumBasics ▼
Answer
For a register-to-register path with the same clock:
Data Arrival Time = T_launch_edge + T_clk_launch (CK latency) + T_ckq + T_comb Data Required Time = T_capture_edge + T_clk_capture (CK latency) − Tsu − T_uncertainty_setup Setup Slack = Data Required Time − Data Arrival Time Where: T_launch_edge = clock edge that launches data (typically 0 for first cycle) T_capture_edge = T_launch_edge + T_period (1 cycle later) T_clk_launch = clock tree delay to launch FF clock pin T_clk_capture = clock tree delay to capture FF clock pin T_ckq = flip-flop clock-to-Q propagation delay T_comb = combinational logic + routing delay Tsu = flip-flop setup time requirement T_uncertainty = clock uncertainty (jitter + guardband)
Positive skew (capture clock later) improves setup: T_clk_capture − T_clk_launch > 0 adds to the window.
Follow-up Questions
  • How does the hold check equation differ?
    Hold: Data Arrival (min path) ≥ Capture_clock_edge + T_clk_capture (min) + Th + T_uncertainty_hold. For hold, use MINIMUM data path delay and MINIMUM launch clock delay (fastest path), and MAXIMUM capture clock delay (slowest capture). Hold slack = Arrival − Required (reversed sign from setup).
  • If clock skew is +0.2ns (positive), how does it affect setup and hold slacks?
    Positive skew = capture clock arrives later. For setup: Required time increases (better setup slack, +0.2ns). For hold: Arrival − Required decreases (worse hold slack, −0.2ns). Positive skew always helps setup, hurts hold. Useful skew exploits this tradeoff.
Q7 What is clock skew? What is positive vs negative skew? EasyBasics ▼
Answer
Clock skew is the spatial difference in clock arrival times at two flip-flops that are supposed to be triggered by the same edge. It arises from different path lengths through the clock tree (buffers, wires).

Skew = T_clk_capture − T_clk_launch

Positive skew (capture later than launch): The capture FF's clock arrives after the launch FF's clock. This helps setup (gives data more time to travel) but hurts hold (data could arrive before the delayed capture edge).

Negative skew (capture earlier than launch): The capture FF's clock arrives before the launch FF's clock. This hurts setup (less time available) but helps hold.
Short Answer
Skew = clock arrival time difference at capture vs launch FF. Positive skew: helps setup, hurts hold.
Follow-up Questions
  • What is useful skew and how is it different from unwanted skew?
    Useful skew is intentionally introduced skew to improve timing — the CTS tool deliberately delays one FF's clock to help a critical path. Unwanted skew arises from unbalanced clock trees and hurts other paths. CTS by default tries to minimize skew (balanced tree); useful skew optimization intentionally imbalances it for timing improvement.
Q8 What is clock jitter and how is it modeled in STA? EasyBasics ▼
Answer
Jitter is the cycle-to-cycle variation in the clock period — the clock edge does not always arrive at exactly T, T+period, T+2×period. It deviates due to: PLL phase noise, power supply noise (VCO sensitivity), electromagnetic interference, substrate noise coupling.

Modeling in STA: Jitter is captured as part of set_clock_uncertainty. The uncertainty value includes:
— PLL output jitter
— On-chip clock tree variation (partially replaced by OCV post-CTS)
— Guardband for measurement/extraction inaccuracy

Typical values: 100–300ps setup uncertainty for a well-characterized PLL. Jitter effectively reduces the available setup window.
Short Answer
Cycle-to-cycle clock period variation; modeled as set_clock_uncertainty which tightens setup slack.
Follow-up Questions
  • What is the difference between random jitter and deterministic jitter?
    Random jitter (RJ) is unbounded Gaussian noise — statistically characterized with sigma. Deterministic jitter (DJ) has a fixed magnitude with specific causes (power supply ripple at a known frequency, crosstalk). Total jitter (TJ) = DJ + some multiple of RJ sigma at a target BER. PLL specs often separate RJ and DJ.
Q9 What is a PVT corner? Which corner is used for setup analysis vs hold analysis? EasyBasics ▼
Answer
PVT = Process + Voltage + Temperature — the three factors that determine cell delay:
— Process: Manufacturing variation (fast/typical/slow transistors)
— Voltage: Supply voltage (higher Vdd = faster cells)
— Temperature: Operating temperature (traditionally, hotter = slower; but see temperature inversion)

Setup analysis corner (worst-case slow): ss_0.72v_125c
Slow process + lowest voltage + highest temperature → slowest cells → hardest to meet setup.

Hold analysis corner (best-case fast): ff_0.88v_-40c
Fast process + highest voltage + lowest temperature → fastest cells → data arrives fastest → most likely to violate hold.
Follow-up Questions
  • What is temperature inversion and which nodes is it relevant to?
    At 28nm and below, at low supply voltages (near-threshold), transistor delay can actually DECREASE with increasing temperature in some operating regions — this is temperature inversion. It means the slowest corner might be cold (ss_0.72v_-40c), not hot. Both hot and cold slow corners must be checked at advanced nodes. This is why modern MCMM setups include multiple temperature points.
  • What is a typical corner and when do you use it?
    tt_0.8v_25c — used for power estimation, functional verification, and early-stage analysis. Not used for sign-off timing because it doesn't capture worst-case margins. However, it's useful for detecting paths that are wildly off even under nominal conditions.
Q10 What is propagation delay and what factors affect it? EasyBasics ▼
Answer
Propagation delay is the time for a signal to travel from a cell's input to its output, measured at 50% of the voltage swing. Two main components:

1. Cell (intrinsic) delay: Time within the cell itself. Depends on: input slew rate (faster input → faster output, lower cell delay), output capacitive load (more load → longer charge time → higher delay), cell drive strength, temperature, voltage, process.

2. Net (interconnect) delay: RC delay of the routing wire. Depends on: wire length and width (resistance), coupling capacitance to adjacent nets, technology node RC values.

In NLDM (Non-Linear Delay Model), cell delay is looked up from a 2D table with axes: input_slew × output_load.
Follow-up Questions
  • How does fanout affect propagation delay?
    High fanout means the driving gate sees higher total load capacitance. More capacitance → more time to charge/discharge → higher cell delay + slower output slew. The slower output slew propagates to downstream cells, increasing their input-slew-dependent delays too. This cascades and is why set_max_fanout and buffer insertion matter.
Q11 What is the critical path and how is fmax derived from STA results? EasyBasics ▼
Answer
The critical path is the timing path with the least (or most negative) setup slack — the longest path in terms of delay relative to the clock period. It is the bottleneck that limits the maximum operating frequency.

fmax from STA:
fmax = 1 / (T_constrained - WNS_setup) Example: Clock period = 1.0ns (1GHz), WNS = -0.15ns → Actual fmax = 1 / (1.0 + 0.15) = 1 / 1.15 ≈ 869 MHz If WNS is positive, the design can run faster than constrained: fmax = 1 / (1.0 - 0.05) = 1 / 0.95 ≈ 1.05 GHz
Follow-up Questions
  • How do you find the critical path in PrimeTime?
    report_timing -delay_type max -max_paths 1 -sort_by slack shows the single worst path. To see top N: report_timing -max_paths 10 -nworst 1 -sort_by slack -delay_type max. The output shows startpoint, endpoint, and the full path trace with individual delays.
Q12 What is a timing arc? What types of timing arcs exist in a flip-flop? EasyBasics ▼
Answer
A timing arc is a defined delay relationship from one pin to another within a cell or between cells in the timing graph. Each arc has characterized delay values in the Liberty library.

Flip-flop timing arcs:
— CK→Q (propagation arc): Clock-to-output delay. How long after the clock edge does Q become valid? Used in arrival time computation.
— D→CK (setup arc): Setup time constraint — D must be stable this long before CK edge.
— D→CK (hold arc): Hold time constraint — D must remain stable this long after CK edge.
— CK→Q (hold arc): For hold analysis, minimum CK→Q delay is used.
— Async pin arcs (RN/SN→Q): Asynchronous reset/set to Q arcs — recovery/removal checks.
— Pulse width arcs: Minimum high/low pulse width on clock pin.
Follow-up Questions
  • What is a negative setup time? Is it physically possible?
    Yes — a negative setup time means data can arrive AFTER the clock edge and still be captured correctly. This occurs in some high-speed, specially designed flip-flops or when clock path internal delays create a window where data can arrive slightly late. It effectively gives extra time but increases Q propagation delay. Rare but valid in Liberty models.
Q13 What is metastability? Can STA fully detect or prevent it? MediumBasics ▼
Answer
Metastability is a physical phenomenon where a flip-flop's output enters an indeterminate voltage state (neither valid 0 nor 1) when the setup or hold time is violated. The bistable storage element has two stable states; when forced into the unstable equilibrium point, it oscillates or drifts before eventually resolving — taking a random amount of time.

Can STA detect or prevent it?
— STA can detect setup/hold violations that could CAUSE metastability within the same clock domain.
— STA CANNOT prevent metastability at asynchronous clock crossings (CDC paths) because by definition there's no timing relationship — some clock edge will always be within the metastability window.
— The solution for CDC is synchronizers (chains of FFs with sufficient MTBF). STA verifies the synchronizer chain timing, not the metastability probability itself.
Short Answer
STA detects violations that CAUSE metastability. Cannot prevent it at CDC — that requires synchronizer design.
Q14 What is the startpoint and endpoint of a timing path? EasyBasics ▼
Answer
Startpoint: Where the data propagation begins. Can be:
— Clock pin (CK) of a flip-flop/latch (reg2reg, reg2out paths)
— Input port of the design (in2reg, in2out paths)

Endpoint: Where the data propagation terminates and the timing check is performed. Can be:
— Data pin (D) of a flip-flop/latch (setup/hold check performed here)
— Output port of the design
— Asynchronous control pin (RN, SN, EN) for recovery/removal checks

The tool reports which startpoint/endpoint pair has the worst timing in each report.
Follow-up Questions
  • Is a net ever a timing path endpoint?
    No — nets are never directly endpoints in STA. Timing checks are performed at pins (cell input pins or output ports), not on nets. However, constraints like set_false_path -through [get_nets X] use nets as waypoints, not endpoints.
Q15 What is slew (transition time) and why does it matter in STA? EasyBasics ▼
Answer
Slew (transition time) is how fast a signal transitions from 0→1 (rise) or 1→0 (fall), measured between 10–90% or 20–80% of the supply voltage. Faster slew = sharper signal edges.

Why it matters:
1. Cell delay increases with slow input slew — NLDM tables show delay worsens as input slew increases. A slow-input cell takes longer to reach its switching threshold.
2. Short-circuit power: Slow transitions cause longer periods when both PMOS and NMOS are simultaneously ON → increased short-circuit current.
3. Signal integrity: Slow slew makes signals more susceptible to crosstalk glitches and noise.
4. DRC check: set_max_transition limits slew to prevent degradation cascading through the design.
Q16 What is a Liberty (.lib) file and what information does it contain for STA? EasyBasics ▼
Answer
A Liberty (.lib) file is the timing and power characterization file for a standard cell library. It contains:

— Cell timing arcs: Propagation delays (NLDM or CCS tables indexed by input slew × output load)
— Flip-flop timing: Setup, hold, recovery, removal, CK→Q delays
— DRC attributes: max_capacitance, max_transition, max_fanout per cell pin
— Operating conditions: PVT point this library represents
— Pin capacitances: Input and output pin capacitance values
— Function: Boolean function of each output pin
— Power: Dynamic and leakage power tables
— AOCV/LVF tables: Advanced OCV or Parametric OCV variation data
Follow-up Questions
  • What is the difference between NLDM and CCS models in Liberty?
    NLDM (Non-Linear Delay Model): 2D tables of delay and output slew vs input_slew × output_cap. Simple, compact, widely supported. CCS (Composite Current Source): Models the cell output as a time-varying current source waveform. More accurate at advanced nodes because it captures load-dependent current behavior, enabling accurate waveform propagation and SI analysis. CCS files are larger but required for sign-off below 28nm.

  • What does -path_type full_clock do differently?
    full_clock expands the entire clock path trace (from clock source, through all clock buffers, to the FF CK pin) in addition to the data path. Without it, clock path is shown as a single lumped "clock tree delay" number. Use full_clock when debugging clock tree issues or unexpected skew.
Q18 What is check_timing and why must you run it before sign-off? EasyBasics ▼
Answer
check_timing is a PrimeTime command that audits the constraint setup for completeness and correctness. It flags:

— Unconstrained paths: Timing paths with no clock — these are skipped silently, hiding potential violations
— Multiple clocks on one pin without proper handling
— Logic loops: Combinational feedback loops that break STA
— No input/output delays on ports
— Mismatched clock definitions
— Ideal network issues

Unconstrained paths are the #1 sign-off risk. A path with no clock appears to have zero slack (or infinite positive slack) — masking a real violation. check_timing output should have zero errors before sign-off.
⚠ Senior Note: In large SoCs, hundreds of unconstrained paths are common in early stages. Each "no clock" must be traced to either: (a) missing create_clock, (b) clock not propagating through a gate, or (c) a genuine non-functional path needing set_false_path.
Q19 What is fanout and what DRC issues does high fanout cause? EasyBasics ▼
Answer
Fanout is the number of input pins driven by a single gate output (or port). High fanout means the driving cell sees a large total capacitive load.

DRC issues from high fanout:
— Slow slew: Large load capacitance → longer charge/discharge time → slow transitions → slew DRC violation
— High cell delay: Driver's delay increases with load — degrades setup timing
— Noise sensitivity: Slow-slew nets are more susceptible to crosstalk glitches
— Power: Large switching capacitance → higher dynamic power

Fix: set_max_fanout 20 [current_design] — synthesis/optimization inserts buffer trees to split high-fanout nets.
Q20 What is clock-to-Q delay (Tco)? How is it different for setup vs hold analysis? EasyBasics ▼
Answer
Clock-to-Q delay (Tco) is the propagation delay from the active clock edge arriving at the FF's CK pin to when the output Q stabilizes at the correct logic value. It is characterized in Liberty as a timing arc from CK to Q, dependent on clock slew and output load.

For setup analysis (max): The STA tool uses the maximum Tco — pessimistic, data arrives as late as possible at the next stage.

For hold analysis (min): The STA tool uses the minimum Tco — optimistic, data arrives as early as possible. If this minimum delay is still too fast for the capture FF's hold requirement, there's a hold violation.

This is why Liberty cells have both cell_rise/fall (max) and timing_type: min_value arcs.
Follow-up Questions
  • Can Tco be 0? What would that imply?
    No — Tco cannot realistically be 0 (it takes some finite time to drive Q). In Liberty, a very small minimum Tco (e.g., 50ps) makes hold violations more likely. Designs with short paths and small minimum Tco are hold-critical. Some theoretical models assume Tco_min = 0 as worst case for hold analysis, which is conservative.
Q21 What is the difference between a latch and a flip-flop in STA context? What is time borrowing? MediumBasics ▼
Answer
Flip-flop (edge-triggered): Captures data on a specific clock edge. Strictly one period of data path delay allowed. No time borrowing possible.

Latch (level-sensitive): Transparent when clock is high (or low), stores when clock goes to opposite level. When transparent, data propagates through continuously.

Time borrowing: In a latch-based pipeline, a stage that is slightly slow can "borrow" time from the next open latch window. If a signal arrives late at a latch, it can propagate through while the latch is still transparent (borrowing from the next cycle). This distributes slack across pipeline stages. Latch-based designs can be more timing-efficient but are harder to analyze and verify.

STA tools handle latches with special timing analysis accounting for the open window and borrowed time.
Follow-up Questions
  • Is time borrowing always safe?
    No. There's a maximum borrow limit — you can't borrow more than the latch open window duration. If total borrowed time exceeds the window, the pipeline stalls or corrupts data. Also, time borrowing can make hold analysis more complex since the window when hold is checked depends on when data actually arrives.
Q22 A 2GHz clock has a period of 0.5ns. How does this constrain your combinational logic budget? EasyBasics ▼
Answer
With a 0.5ns period, the setup timing equation gives:
Available logic time ≈ T_period + Skew − Tco_max − Tsu − T_uncertainty Typical numbers at 7nm: T_period = 0.5 ns Tco_max ≈ 0.05–0.10 ns Tsu ≈ 0.03–0.06 ns Uncertainty ≈ 0.05–0.10 ns Net routing ≈ variable → Logic budget ≈ 0.5 - 0.10 - 0.05 - 0.10 ≈ 0.25 ns → Roughly 4–6 FO4 gate delays at 7nm
This is extremely tight — requires 2–3 gate logic depth max. High-performance microprocessors at these speeds use aggressive pipelining, custom cells, and extensive floorplanning.
Follow-up Questions
  • What is FO4 (Fan-Out-of-4) delay and why is it used as a metric?
    FO4 is the delay of an inverter driving 4 copies of itself — a technology-independent unit of logic delay. It scales with process node (roughly 10–15ps at 7nm, ~40–50ps at 65nm). It's used to compare logic depth across different technologies and gives a sense of "how many gate delays fit in one clock cycle."
Q23 What's the difference between report_timing, report_timing_summary, and report_qor in PrimeTime? MediumBasics ▼
Answer
report_timing: Full detailed path report. Shows cell-by-cell trace with delays, slews, arrival times. Use when debugging a specific violation. Command: report_timing -path_type full -delay_type max -max_paths 10

report_timing_summary: Concise table showing WNS, TNS, and failing endpoint count per path group. Use for quick project status. Command: report_timing_summary

report_qor (Quality of Results): High-level summary of timing, design rules, area, and optimization status across all path groups and scenarios. Includes DRC summary. Most useful for daily sign-off tracking. Command: report_qor
Follow-up Questions
  • How do you filter report_timing to show only negative slack paths?
    report_timing -slack_lesser_than 0 -delay_type max -max_paths 9999 — this reports all paths with slack less than 0 (violating paths only). Combine with -path_type summary for a condensed view of all failures.
Q24 What are recovery and removal timing checks? MediumBasics ▼
Answer
These checks apply to asynchronous control pins (async reset RN, async set SN, async enable):

Recovery time: Minimum time the asynchronous control signal (e.g., async reset) must be de-asserted (released) BEFORE the next active clock edge. Analogous to setup time but for async pins. Ensures the FF is fully out of reset state before the next data capture.

Removal time: Minimum time the asynchronous control signal must remain asserted AFTER the active clock edge before being de-asserted. Analogous to hold time for async pins. Ensures the async function fully takes effect before being removed.

Violations cause metastability just like setup/hold violations. These are checked in STA by default if the library characterizes them.
Follow-up Questions
  • How do you ensure recovery/removal checks pass for an async reset?
    Options: (1) Set_false_path if reset is never de-asserted close to a clock edge (true async reset, never deasserted during normal operation). (2) If recovery matters, constrain the reset de-assertion path as a real timing path. (3) Use a reset synchronizer to de-assert reset synchronously.
Q25 What is a half-cycle path? Where does it commonly appear? MediumBasics ▼
Answer
A half-cycle path is a timing path where data is launched on one clock edge (e.g., rising) and captured on the opposite clock edge (e.g., falling) of the same clock. The available setup window is only half a clock period instead of a full period.

Setup equation for half-cycle path:
Available time = T_period / 2 + Skew − Tco − Tsu − uncertainty
Where half-cycle paths appear:
— DDR interfaces (data valid on both edges)
— Mixed positive/negative edge-triggered domains
— Latch-to-FF or FF-to-latch paths
— Deliberately pipelined designs with 2-phase clocking

These paths are inherently more difficult to close and require aggressive optimization.
Follow-up Questions
  • How does the tool identify a half-cycle path vs a full-cycle path?
    The tool examines the launch edge and capture edge waveforms. If the launch is on the rising edge and capture is on the falling edge of the same clock (period 1.0ns), the available window is 0.5ns. The tool identifies this from the clock waveform definition in create_clock -waveform {0 0.5} — the falling edge is at 0.5ns.

Q26 Write the create_clock command for a 1GHz clock with 40% duty cycle on port CLK. EasyClocks ▼
Answer
# 1GHz = 1.0ns period. 40% duty cycle: high for 0.4ns, low for 0.6ns create_clock -name CLK_CORE -period 1.0 -waveform {0 0.4} [get_ports CLK] # -waveform {rise_time fall_time}: rising edge at 0ns, falling at 0.4ns # Default (50%): -waveform {0 0.5} or just -period 1.0 # 500MHz 50%: create_clock -name CLK -period 2.0 [get_ports CLK]
Short Answer
-waveform {0 0.4} means rise at t=0, fall at t=0.4ns for a 1.0ns period clock (40% duty cycle).
Follow-up Questions
  • What does -add flag do in create_clock?
    -add appends a second clock to the same source pin without overwriting the first. Used when a single port legitimately carries two clocks (e.g., a test clock and a functional clock on the same physical pin). Without -add, the second create_clock overwrites the first. Use with -master_clock when multiple clocks share a pin and created_generated_clock needs to know which master to use.
  • What happens if you don't add set_clock_latency or set_clock_uncertainty after create_clock?
    The clock is ideal — zero source latency, zero network latency, zero uncertainty, zero slew. All paths will have over-optimistic timing. Pre-CTS STA without latency/uncertainty is not representative of real-silicon behavior. Always add these immediately after create_clock in a pre-CTS SDC.
Q27 How do you define a PLL output clock that multiplies a 100MHz input by 10x? MediumClocks ▼
Answer
# Master clock on PLL input create_clock -name CLK_REF -period 10.0 [get_ports CLK_REF] # PLL output: 100MHz × 10 = 1GHz (period = 1.0ns) create_generated_clock \ -name CLK_PLL_1G \ -source [get_pins PLL_INST/CLKIN] \ -multiply_by 10 \ [get_pins PLL_INST/CLKOUT] # Model PLL internal latency (phase offset from PLL circuit) set_clock_latency -source 0.5 [get_clocks CLK_PLL_1G]
Follow-up Questions
  • What happens if you forget -master_clock when two clocks arrive at PLL/CLKIN?
    The tool picks one master arbitrarily. If the wrong master is selected, the generated clock will have an incorrect period/phase relationship, leading to either over-optimistic or over-pessimistic inter-clock timing checks. Always specify -master_clock CLK_REF explicitly when multiple clocks can reach the source pin.
  • When is a generated clock still ideal (not propagated)?
    By default, generated clocks are ideal unless set_propagated_clock is applied. Post-CTS, you must run set_propagated_clock [all_clocks] to switch all clocks (including generated) to use actual clock tree delays. Forgetting this leaves generated clocks with zero internal skew, hiding real timing issues.
Q28 Explain the difference between source latency and network latency in set_clock_latency. MediumClocks ▼
Answer
Source latency (-source flag): Delay from the actual clock origin (oscillator, PLL) to the clock definition point in your SDC (the port/pin where you placed create_clock). Models: board PCB trace delay, off-chip driver delay, connector delay, clock buffer chain before the chip.

Network latency (no flag): Delay from the clock definition point (create_clock source) to the flip-flop's clock pin through the internal chip clock tree.

Post-CTS behavior: After CTS, set_propagated_clock replaces network latency with actual propagated delays. Source latency should remain because the board delay hasn't changed.
# 0.3ns board trace + connector to chip CLK pin set_clock_latency -source 0.3 [get_clocks CLK_SYS] # 0.6ns estimated internal clock tree (used pre-CTS only) set_clock_latency 0.6 [get_clocks CLK_SYS]
⚠ Senior Note: Forgetting source latency makes timing optimistic. Missing it means the tool doesn't account for the board delay in inter-chip timing calculations, making I/O constraints incorrect.
Q29 What does set_clock_uncertainty model and what values are typical? MediumClocks ▼
Answer
set_clock_uncertainty is a lump-sum margin that accounts for three things:

1. Clock jitter: Cycle-to-cycle PLL output variation (typically 50–150ps peak-to-peak at modern nodes)
2. Pre-CTS clock tree uncertainty: Expected post-CTS clock skew (replaced by actual skew after CTS)
3. Guardband: Extra margin for extraction inaccuracy, OCV not captured elsewhere

Typical values:
# Pre-CTS: uncertainty includes both jitter + expected skew budget set_clock_uncertainty -setup 0.250 [get_clocks CLK_SYS] # 250ps set_clock_uncertainty -hold 0.100 [get_clocks CLK_SYS] # 100ps # Post-CTS: uncertainty is just jitter (skew is now real) set_clock_uncertainty -setup 0.100 [get_clocks CLK_SYS] set_clock_uncertainty -hold 0.050 [get_clocks CLK_SYS]
Follow-up Questions
  • How does uncertainty interact with CPPR?
    Clock uncertainty is applied globally to every path. CPPR (Clock Path Pessimism Removal) only removes double-derating on the common clock path segment — it does NOT remove uncertainty. Uncertainty always reduces effective timing margin. Only CRPR/CPPR removes structural pessimism from OCV derating on shared path segments.
Q30 What happens if you forget to run set_propagated_clock after CTS? MediumClocks ▼
Answer
Without set_propagated_clock, the tool continues using the pre-CTS ideal (lumped) clock latency values from set_clock_latency, ignoring the actual routed clock tree delays.

Consequences:
— Clock skew is hidden — all FFs appear to have the same clock arrival time (equal latency), giving false zero-skew analysis
— Setup slack is artificially optimistic — real negative skew paths appear to have no penalty
— Hold violations are missed — real negative skew in the capture path (where capture arrives late and launch is early) is invisible
— Sign-off is invalid — the analysis doesn't represent silicon behavior
Short Answer
Clocks remain ideal post-CTS → zero skew illusion → optimistic timing → missed violations → silicon failures.
Q31 When do you use set_clock_groups -asynchronous vs -logically_exclusive vs -physically_exclusive? MediumClocks ▼
Answer
-asynchronous: The two clock groups have no phase relationship (different sources). Paths between groups are NOT timed. Used for true CDC between clocks from different PLLs, oscillators, or sources. Most common option.

-logically_exclusive: Clocks can't be active simultaneously (e.g., mux-selected clocks where only one is active at a time). Paths are NOT timed. However, clock tree buffers can be shared in synthesis/CTS because only one is ever active.

-physically_exclusive: Clocks never coexist even in the physical sense — no shared buffers. Stronger than logically_exclusive. Used for clocks that drive completely separate sub-designs that are never assembled together.
# True asynchronous domains set_clock_groups -asynchronous \ -group [get_clocks CLK_CORE] \ -group [get_clocks {CLK_USB CLK_PCIe}] # Mux-selected test/functional clocks set_clock_groups -logically_exclusive \ -group [get_clocks CLK_FUNC] \ -group [get_clocks CLK_SCAN]
Follow-up Questions
  • How is set_clock_groups -asynchronous different from set_false_path -from clk_a -to clk_b?
    Functionally similar — both suppress timing analysis. Key differences: set_clock_groups is bidirectional by default, named (removable by name), cleaner at scale. set_false_path gives more granularity (-through, -rise, -fall, per-direction). Preference: use set_clock_groups for domain-level CDC; use set_false_path for specific path-level exceptions within the same domain.
Q32 What is CPPR/CRPR and why is it important with OCV enabled? HardClocks ▼
Answer
CPPR (Clock Path Pessimism Removal), also called CRPR, addresses a specific form of OCV over-pessimism:

When launch and capture flip-flops share a common clock path segment (same buffers up to some divergence point), OCV derating applies:
— To the common segment as seen through the launch path: late derate (1.05)
— To the same common segment as seen through the capture path: early derate (0.95)

But those are the same physical cells with one real delay — they can't simultaneously be 5% slower and 5% faster. This is double pessimism.

CPPR correction:
CPPR credit = (Late derated common path delay) - (Early derated common path delay) = common_path_delay × (late_factor - early_factor) = 0.5ns × (1.05 - 0.95) = 0.05ns This 0.05ns is ADDED BACK to setup slack.
Most STA tools apply CPPR automatically when OCV is enabled. Without it, slack is over-pessimistic by 0.05–0.3ns depending on common path length.
Follow-up Questions
  • How do you verify CPPR is being applied in PrimeTime?
    In report_timing -path_type full, look for a "clock reconvergence pessimism removal" line with a positive credit value near the bottom of the timing report. Also check: report_timing -path_type full_clock shows where launch and capture paths diverge. set_app_var timing_crpr_enable true enables CPPR in PT (it may be default ON).
Q33 What is set_clock_transition and when should it be used? MediumClocks ▼
Answer
set_clock_transition specifies the rise/fall slew rate at clock pins for ideal (non-propagated) clocks. Used exclusively pre-CTS when real clock tree buffer slews are unknown.

Without this, ideal clocks have zero slew — unrealistically fast. The flip-flop timing tables are sensitive to clock input slew, so a realistic slew estimate is needed for accurate pre-CTS Tsu, Th, and Tco lookups.

Post-CTS: Becomes irrelevant once set_propagated_clock is applied — propagated delays include real slew values from actual clock buffers.
set_clock_transition 0.15 [all_clocks] # 150ps all clocks set_clock_transition -rise 0.12 -fall 0.18 [get_clocks CLK_DDR]
⚠ Too-optimistic slew (e.g., 0.05ns when real tree gives 0.20ns) under-estimates FF setup/hold times, creating a false timing closure pre-CTS that fails post-CTS.
Q34 When and why would you use set_clock_sense -stop_propagation? MediumClocks ▼
Answer
set_clock_sense -stop_propagation tells the STA tool to terminate clock propagation at a specific pin — the clock does NOT pass beyond that point. Use cases:

1. Test mux bypass: A test mux passes clock in test mode but should not propagate functional clock into the scan chain in functional analysis
2. Clock gating cell where gate is always enabled: No need to propagate clock sense through an always-transparent clock gate
3. Isolating unrelated logic that happens to sit on a clock network

Without it, the tool might incorrectly propagate a clock through combinational logic that isn't a real clock path, creating spurious clock domains on data pins.
set_clock_sense -stop_propagation -clock CLK_FUNC [get_pins SCAN_MUX/Y]

Q36 What is useful skew? What are its benefits and risks? HardClocks ▼
Answer
Useful skew is intentional clock skew inserted during CTS to improve timing margins. Unlike unwanted skew (a side effect of unbalanced trees), useful skew is a deliberate optimization technique:

How it works: For a setup-critical reg2reg path, delay the capture FF's clock (positive skew). Available setup time = T_period + skew. If skew = 0.1ns, it's like adding 100ps to the clock period just for that path.

Benefits:
— Fixes setup violations without changing the logic
— No area impact (just clock routing change)
— Can improve many paths simultaneously if clock domain is skewed uniformly

Risks:
— Positive skew degrades hold: capture clock later → data must arrive later → path needs more hold margin
— Can create hold violations on nearby paths
— Skew is fixed in the implementation — inflexible for future ECOs
— Needs careful management to avoid "skew storms" where fixing one path cascades to new hold violations
Q37 What is a clock gating check and what does a violation mean in silicon? HardClocks ▼
Answer
A clock gating check verifies that the enable signal of an Integrated Clock Gate (ICG) cell is stable during the clock's active window:

Setup (gating check): The EN signal must be stable before the clock's evaluation edge (typically the rising edge when clock is high-active). If EN is still changing when the gate window opens, a glitch can propagate to the gated clock output.

Hold (gating check): EN must remain stable after the clock edge to ensure a clean gate.

Silicon impact of violation: A glitch on the gated clock can cause:
— Spurious clock pulses → flip-flops capture wrong data
— Clock tree DRC issues (too-narrow pulse)
— Power glitches and noise
— These are extremely hard to debug in silicon

The check is separate from data path setup/hold — it's a dedicated timing check type in the tool. Library ICG cells have built-in characterization for this check.
# Override clock gating check timing when library values need adjustment set_clock_gating_check -setup 0.2 -hold 0.1 set_clock_gating_check -setup 0.3 [get_cells PD_A/ICG_*] # per domain
Q38 How do you use -edges in create_generated_clock? Give a divide-by-3 example. HardClocks ▼

-edges {e1 e2 e3} maps master clock edges (numbered from 1 = first rising) to generated clock edges (rising, falling, next rising). This gives direct control over non-standard divisions.

Q39 What is a pulse width check and when does it commonly fail? HardClocks ▼
Answer
A pulse width check verifies that the clock's high and low duration each meet the minimum required values from the Liberty library (min_pulse_width_high and min_pulse_width_low per cell).

Common failure scenarios:
1. Non-50% duty cycle clocks after clock gating: ICG cells can skew the output duty cycle, narrowing pulses
2. Clock dividers with non-symmetric output (e.g., div-by-3 without 50% duty)
3. After large clock tree delays: Clock path inversion or mismatch can narrow pulses at leaf nodes
4. At advanced corners (ff, high Vdd): Clock buffers are faster, transition time is shorter relative to period
5. Multi-phase clocking with aggressive phase offsets

Fix: Add clock buffers to restore slew, adjust duty cycle correction circuit, restructure the gating logic, or use a duty cycle corrector (DCC) in the clock tree.
Q40 How do you model different clock uncertainty for inter-clock (CDC) paths? HardClocks ▼

# Intra-clock uncertainty (paths within same clock domain) set_clock_uncertainty -setup 0.15 [get_clocks CLK_A] set_clock_uncertainty -setup 0.15 [get_clocks CLK_B] # Inter-clock uncertainty (paths from CLK_A domain to CLK_B domain) # Higher uncertainty because two independent PLLs with uncorrelated jitter set_clock_uncertainty \ -from [get_clocks CLK_A] \ -to [get_clocks CLK_B] \ 0.40 # Edge-specific: only rising-to-rising crossing for DDR set_clock_uncertainty \ -rise_from [get_clocks CLK_DDR] \ -fall_to [get_clocks CLK_DDR] \ 0.10
Inter-clock uncertainty accounts for the fact that two independently-generated clocks have uncorrelated jitter — their relative phase uncertainty is the sum of individual jitters (worst case), not just one clock's jitter.

Q42 What is different about STA strategy pre-CTS vs post-CTS? What changes in the SDC? ExpertClocks ▼
Answer
Pre-CTS STA:
— Clocks are ideal (no physical tree yet)
— Use set_clock_latency (network + source), set_clock_transition, set_clock_uncertainty (includes skew budget)
— OCV applied only to data paths (or with reduced clock derating)
— Goal: ensure logic is synthesized to meet timing under expected post-CTS conditions
— Higher uncertainty margin needed to budget for unknown real clock skew

Post-CTS STA:
— Replace set_clock_latency (network) with set_propagated_clock [all_clocks]
— Reduce set_clock_uncertainty to jitter only (skew is now real, not guessed)
— Apply full OCV to both clock and data paths
— CPPR enabled automatically
— Source latency retained
— This is the mode for sign-off

SDC typically has two versions: a pre-CTS version (with latency/uncertainty) and a post-CTS version (with propagated clock). Synthesis uses pre-CTS SDC; final PnR sign-off uses post-CTS SDC.

Q44 What is clock reconvergence (not CPPR) in the data path and how does it affect STA accuracy? ExpertClocks ▼
Answer
Reconvergent fanout in the DATA path occurs when a signal splits into two branches and the branches re-merge at a logic gate downstream. STA handles each branch independently and takes the worst case at the merge point.

The problem: The two branches from the same source are correlated — they both experience the SAME process variation and transition. STA, however, treats them independently, potentially applying opposing OCV derates (one late, one early), which is physically impossible for signals from the same source.

Impact: This can be pessimistic (tool applies worst-case delay on both branches independently even when they can't both be worst-case simultaneously). Modern tools handle this with ATIS (Arrival Time with Independent Slews) or graph-based timing that tracks signal identity through reconvergence points.

Practical impact: For most designs, this is a minor effect. For paths with significant reconvergent fanout and high OCV margins, it can add 10–50ps of false pessimism.
Q45 A new engineer asks: "My design passes timing pre-CTS but fails post-CTS. Why?" Explain the reasons. HardClocks ▼
Answer
Several real factors change between pre-CTS and post-CTS:

1. Real clock skew appears: Pre-CTS used an estimated uniform latency. Post-CTS, real skew can be negative (capture arrives earlier than launch) → tightens setup; real positive skew creates hold violations.

2. Uncertainty reduced but real skew larger than estimated: If the clock tree is poorly balanced, real skew > pre-CTS uncertainty budget → paths that seemed safe now fail.

3. Clock tree insertion delay: Real clock buffers have actual delay. If the network latency estimate was too optimistic (too small), capture clock arrives later than expected for setup, reducing the data budget.

4. Full OCV now applied to clock paths: Pre-CTS often had reduced or no clock path OCV. Post-CTS, full OCV on clock paths tightens margins further.

5. New hold violations: Post-CTS, short paths that were fine with ideal clocks now show hold violations due to real clock tree mismatches.

6. Clock slew changes from set_clock_transition: Real clock tree delivers different slew than estimated, changing Tsu/Th/Tco values.
Q46 What are the primary goals of Clock Tree Synthesis (CTS)? MediumClocks ▼
Answer
Primary CTS goals (in priority order for most designs):

1. Minimize clock skew: All FFs in a domain should see the clock at approximately the same time. Skew directly impacts timing margin.

2. Meet insertion delay budget: Total latency from clock source to FF CK pins should stay within the allocated budget to preserve setup margin.

3. Meet slew requirements: Clock signal at every CK pin must have slew within set_max_transition -clock_path limit.

4. Minimize power: Clock network is often 30–40% of total dynamic power. Use low-power CTS (clock gating, hybrid trees, fewer buffers).

5. Useful skew optimization (advanced): Introduce beneficial skew to fix timing without additional logic changes.

6. DRC compliance: No max_cap/transition violations on any clock net.
Q47 Write the complete SDC for a DDR4 data bus (both edges) coming into a chip. HardClocks ▼

Q49 Clock trees consume up to 30–40% of dynamic power. What techniques reduce clock power? ExpertClocks ▼
Answer
1. Integrated Clock Gating (ICG): Gating the clock to inactive register banks eliminates dynamic power for those FFs. Most impactful technique — can reduce total clock power by 20–60% in data-stationary designs.

2. Multi-level clock gating: Gate at both register level and module level (coarser granularity saves more power when large blocks are inactive).

3. Clock domain isolation: Power-down entire clock domains (clock shutdown) for blocks that are power-gated.

4. Minimum-buffer CTS: Reduce the number of clock buffers by allowing slightly more skew — traded for power. H-tree topologies can be more efficient than fishbone for uniform cell distribution.

5. Low-power CTS cells: Use library cells with lower drive strength where the load allows, reducing short-circuit current.

6. Frequency reduction: Clock gating at module level, lower frequency for sub-blocks (DVFS).

7. Resonant clocking: Advanced technique using inductors to recycle clock energy. Used in server processors.
Q50 What is a clock mesh vs a clock tree? When would you choose a clock mesh? ExpertClocks ▼
Answer
Clock tree: Hierarchical buffer tree from clock source to FFs. Good skew control with careful design. Compact routing. Standard for most designs.

Clock mesh (grid): A distributed resistive mesh of wires across the chip, with multiple driver points feeding the grid. FFs connect to the nearest mesh node.

Clock mesh advantages:
— Extremely low skew (mesh averages out local delays) — typically <20ps compared to 50–200ps for trees
— Very robust to process variation — mesh impedance averages out local variations
— Better SI noise immunity (distributed low-impedance clock)

Clock mesh disadvantages:
— Very high power: mesh is always active, large RC → high switching power
— Large routing area (full-chip metal grid)
— Complex design and analysis

Use cases: High-performance microprocessors (Intel, AMD), designs requiring ultra-low skew at GHz+ frequencies, custom digital circuits at advanced nodes.

Q51 What are launch edge and capture edge? For a 1GHz clock, what are the default edges for setup and hold? EasyPaths ▼
Answer
Launch edge: The clock edge that causes the launch FF to send (launch) data from its Q output. Data starts propagating through combinational logic from this moment.

Capture edge: The clock edge at the capture FF that samples the incoming data. This is when the setup/hold check is performed.

For a 1GHz clock (T = 1.0ns):
— Setup check: Launch at t=0ns, capture at t=1.0ns (one full period later). The data has 1.0ns (minus Tsu, Tco, uncertainty) to travel.
— Hold check: Both launch and capture happen at the SAME edge (t=0ns). The data from the launch FF must NOT arrive at the capture FF before the same-cycle edge completes its capture. Hold check uses minimum delays.
Follow-up Questions
  • For a multicycle path of 2 (setup N=2), where is the capture edge?
    The capture edge moves to t=2.0ns (2 periods after launch). The hold check must also be adjusted with set_multicycle_path -hold 1 to move the hold reference from t=0 to t=1.0ns (one period behind the new capture edge). Without -hold 1, hold is checked at t=0 — causing phantom hold violations.
