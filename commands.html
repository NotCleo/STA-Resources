Clock Definition & Control

8 commands
create_clock Clock
Define a primary (master) clock on a port or pin
create_clock [-name <name>] [-period <period>] [-waveform <waveform_list>] [-add] [-comment <string>] [source_objects]
Purpose
Creates a timing clock object and anchors it to a port or pin. The STA tool derives all setup/hold checks from this clock's period and waveform.
Effect on STA
All paths clocked by this object are constrained with setup (max) and hold (min) checks. No clock = no timing check on that path.
FlagTypeDescription
-namestringLogical clock name; defaults to source port/pin name. Use explicit names for clarity in reports.
-periodfloat (ns)Full clock period. This is the setup window. Mandatory unless waveform fully implies it.
-waveformlist of floatsList of rising and falling edge times within one period. Default is {0 period/2}. E.g., {0 3 5 8} defines a non-50% duty cycle clock with 2 edges per half-period.
-addflagAppend a second clock to the same source without overwriting the first. Used for paths with multiple clocking domains on one pin (rare, use carefully).
source_objectscollectionPort(s) or pin(s) where the clock originates. Use get_ports or get_pins.
Examples
# 1GHz clock, 50% duty cycle, on port CLK create_clock -name CLK_CORE -period 1.0 -waveform {0 0.5} [get_ports CLK] # 40% duty cycle, 500MHz clock create_clock -name CLK_DDR -period 2.0 -waveform {0 0.8} [get_ports DDR_CLK]
⚠ Senior Note: An ideal clock has zero latency and zero uncertainty by default. Always add set_clock_latency and set_clock_uncertainty after create_clock for realistic pre-CTS analysis. Missing these makes sign-off optimistic.
create_generated_clock Clock
Define a clock derived from another clock (PLL output, clock divider, mux output)
create_generated_clock [-name <name>] -source <master_pin> [-master_clock <name>] [-divide_by <int>] [-multiply_by <int>] [-duty_cycle <pct>] [-invert] [-edges <list>] [-edge_shift <list>] [-combinational] [-add] [-preinvert] [source_objects]
Purpose
Declares a clock that is topologically derived from a master clock. The relationship preserves phase/frequency information for inter-domain timing checks.
Effect on STA
The tool can analyze synchronous relationships between master and generated clocks. Without this, crossing paths become asynchronous (unconstrained or false-pathed).
FlagTypeDescription
-sourcepinPin from which this clock is derived (e.g., PLL input, clock divider input). This is the clock source point — required.
-master_clockclock nameExplicitly specify master clock at the source pin when multiple clocks exist there.
-divide_byintegerGenerated clock period = master_period × divide_by. E.g., -divide_by 2 halves the frequency.
-multiply_byintegerGenerated clock period = master_period / multiply_by. Used for PLL output that multiplies frequency.
-duty_cyclefloat %Override the duty cycle of the generated clock (0–100). Useful for non-symmetric dividers.
-invertflagGenerated clock is phase-inverted relative to master. Used for falling-edge triggered domains.
-edgeslistExplicit edge mapping from master to generated. E.g., {1 3 5} means rising edge of generated comes from master edge 1, falling from edge 3, next rising from edge 5.
-edge_shiftlist (ns)Time shift applied to each edge in -edges list. Handles PLL phase offset.
-combinationalflagThe path from source to this clock is purely combinational (no flip-flop). Used for mux-selected clocks.
Examples
# PLL output: 4x multiplied clock create_generated_clock -name CLK_PLL_OUT -source [get_pins PLL/CLKIN] -multiply_by 4 [get_pins PLL/CLKOUT] # Clock divider: div-by-2 create_generated_clock -name CLK_DIV2 -source [get_ports CLK] -divide_by 2 [get_pins U_DIV/Q] # Clock mux output (combinational path) create_generated_clock -name CLK_MUX_OUT -source [get_pins CLK_MUX/I0] -combinational [get_pins CLK_MUX/Y]
⚠ Senior Note: Forgetting -master_clock when multiple clocks share a source pin causes the tool to pick arbitrarily. Generated clocks must be re-propagated with set_propagated_clock post-CTS, or they remain ideal.
set_clock_groups Clock
Define timing relationships between clock domains (CDC handling)
set_clock_groups [-name <name>] [-asynchronous | -logically_exclusive | -physically_exclusive] [-allow_paths] -group <clocks> -group <clocks> …
Purpose
Tells the timing tool how to treat paths between clock groups. Equivalent to set_false_path -from clk1 -to clk2 for asynchronous domains, but more semantically correct and manageable at scale.
Effect on STA
Paths between specified groups are either ignored (async/exclusive) or kept for analysis. Prevents false violations on legitimate CDC paths. With -allow_paths, violations are reported but excluded from critical paths.
FlagDescription
-asynchronousClocks have no phase relationship. Paths between groups are NOT timed. Most common for CDC. Equivalent to bidirectional false paths.
-logically_exclusiveClocks cannot be active simultaneously (e.g., mux-selected clocks). Paths not timed, but clock tree buffers are shared in synthesis.
-physically_exclusiveClocks cannot even physically coexist. No shared buffers. Stronger than logically_exclusive.
-allow_pathsStill analyze and report paths but do not count them toward critical paths in optimization.
-group {clocks}Repeatable. Each -group specifies a set of clocks forming one domain. Can pass multiple clocks per group.
Examples
# Fully asynchronous domains — no CDC timing analysis set_clock_groups -asynchronous -group [get_clocks CLK_SYS] -group [get_clocks CLK_USB] # Mux-selected clocks (logically exclusive) set_clock_groups -logically_exclusive -group [get_clocks CLK_FAST] -group [get_clocks CLK_SLOW]
💡 In Synopsys PrimeTime, -asynchronous is the most frequently used option. Use -name so you can identify and delete individual groups later with remove_clock_groups.
set_clock_latency Clock
Specify source or network latency for a clock (used pre-CTS)
set_clock_latency [-rise | -fall] [-min | -max] [-source] [-early | -late] [-clock <clock_name>] latency_value object_list
Purpose
Models the delay from clock source (e.g., off-chip oscillator) to the clock definition point (-source latency) and from the clock definition point to the flip-flop clock pin (network latency). Essential for accurate pre-CTS timing.
Effect on STA
Source latency shifts the clock waveform at the definition point. Network latency shifts the clock arrival at register CPs. Both affect setup and hold checks. After CTS, network latency is replaced by propagated delays.
FlagDescription
-sourceSpecifies source latency (board trace, PCB, off-chip delay from oscillator to the ASIC input pin). If omitted, applies network latency (intra-chip clock tree).
-rise/-fallApply latency only to rising or falling edge of the clock, respectively.
-min/-maxApply to min (hold) or max (setup) analysis corners. Skipping applies to both.
-early/-lateUsed in MMMC. Applies to early (hold) or late (setup) timing arc analysis.
Examples
# 0.5ns board delay before chip input, 0.8ns clock tree inside set_clock_latency -source 0.5 [get_clocks CLK_SYS] set_clock_latency 0.8 [get_clocks CLK_SYS] # Min/Max latency for hold/setup separately set_clock_latency -source -min 0.3 [get_clocks CLK_SYS] set_clock_latency -source -max 0.7 [get_clocks CLK_SYS]
⚠ Senior Note: After CTS, replace set_clock_latency (network) with set_propagated_clock to use actual routed delays. Source latency should remain if the board delay model is needed.
set_clock_uncertainty Clock
Model clock jitter, skew, and timing margin uncertainty
set_clock_uncertainty [-rise | -fall] [-setup | -hold] [-from <clock> [-rise_from|-fall_from]] [-to <clock> [-rise_to|-fall_to]] uncertainty_value [object_list]
Purpose
Accounts for clock uncertainty: PLL jitter, on-chip variation in clock tree, measurement inaccuracy. Applied as a timing margin on top of computed skew. The tool subtracts uncertainty from setup slack and adds it to hold slack.
Effect on STA
Setup uncertainty tightens available time window. Hold uncertainty makes hold check harder to satisfy. For inter-clock paths, the -from/-to version allows different uncertainty per crossing direction.
FlagDescription
-setupApply uncertainty only to setup (max delay) check. Typical: 100–300ps for PLL jitter + OCV margin.
-holdApply uncertainty only to hold (min delay) check.
-from/-toSpecify inter-clock uncertainty. Must be used together. Used for clock domain crossing setup/hold uncertainty.
-rise_from/-fall_to etc.Edge-specific crossing uncertainty for fine-grained CDC modeling.
Examples
# 200ps setup uncertainty, 100ps hold uncertainty on CLK_SYS set_clock_uncertainty -setup 0.2 [get_clocks CLK_SYS] set_clock_uncertainty -hold 0.1 [get_clocks CLK_SYS] # Inter-clock uncertainty for CDC path set_clock_uncertainty -from [get_clocks CLK_A] -to [get_clocks CLK_B] 0.4
set_clock_transition Clock
Set rise/fall transition time on clock ports/pins for ideal clocks
set_clock_transition [-rise | -fall] [-min | -max] [-clock <clock>] transition_value clock_list
Purpose
Defines how fast the clock edge transitions (slew rate). Affects cell timing since most LUT/cell delays are functions of input slew. Used pre-CTS when real slew from clock tree buffers is unknown.
Effect on STA
The tool uses this value when looking up clock pin timing tables. A slow clock transition increases cell delay and D-to-Q delay. Overrides propagated slew for ideal networks.
Examples
# Set 150ps rise/fall transition on all clocks (pre-CTS estimation) set_clock_transition 0.15 [all_clocks] set_clock_transition -rise 0.12 -fall 0.18 [get_clocks CLK_DDR]
⚠ Senior Note: After CTS with set_propagated_clock, set_clock_transition is overridden by propagated values. Keep it for pre-CTS only. Overly optimistic slew gives unconservative timing estimates.
set_propagated_clock Clock
Switch from ideal to propagated (actual) clock network delays
set_propagated_clock object_list
Purpose
Instructs the STA tool to compute actual clock arrival times by propagating through the physical clock tree network (buffers, wires) rather than using ideal lumped latency values.
Effect on STA
Clock skew is now computed from actual network delays. This is the correct mode for post-CTS sign-off. It overrides set_clock_latency (network) but NOT source latency.
Examples
# Enable propagated clocks for all clocks — standard post-CTS SDC set_propagated_clock [all_clocks] # Selective propagation for a specific clock set_propagated_clock [get_clocks CLK_CORE]
💡 Use all_clocks to apply to every clock in one line. In a multi-mode setup, apply per-scenario. Ideal clocks after CTS will have over-optimistic slack because skew is hidden.
set_clock_sense Clock
Override clock propagation sense at a specific pin (Synopsys/Cadence specific)
set_clock_sense [-positive | -negative | -stop_propagation] [-clock <clock>] [-pulse rising_or_falling] pin_list
Purpose
Tells the tool whether a given pin sees a non-inverted (+), inverted (-), or no clock propagation through it. Used to handle complex clock gating cells where automatic sense inference fails.
Effect on STA
-stop_propagation terminates clock propagation at that pin — downstream is not clocked by this clock. -negative tells the tool this pin sees an inverted clock. Incorrect sense can cause wrong setup/hold checks.
Examples
# Stop CLK from propagating through a test mux output set_clock_sense -stop_propagation -clock CLK_SYS [get_pins TEST_MUX/Y] # Force negative sense (inverted clock domain) set_clock_sense -negative -clock CLK_SYS [get_pins INV_CLK_BUF/Y]

Input / Output Delay Constraints

2 commands
set_input_delay I/O
Model external combinational delay arriving at an input port relative to a clock
set_input_delay [-max | -min] -clock <clock_name> [-clock_fall] [-rise | -fall] [-add_delay] [-network_latency_included] [-source_latency_included] delay_value port_list
Purpose
Tells the STA tool how much time has already been consumed by the external source driving this input pin. The remaining time in the clock cycle is available for the internal logic.
Effect on STA
Available time = period - setup_time - input_delay (for max). A larger input_delay = less internal slack. The -max variant constrains setup; -min constrains hold.
FlagDescription
-maxMaximum external delay. Used for setup analysis. Models worst-case arrival from external chip.
-minMinimum external delay. Used for hold analysis. Models best-case (fastest) arrival.
-clockReference clock for the delay. The delay is measured relative to this clock's active edge.
-clock_fallThe reference is the falling edge of the clock. Used for DDR interfaces where data is sampled on both edges.
-add_delayDo not override existing delay — add another constraint. Used when the same port is driven by multiple paths with different clocks.
-network_latency_includedThe delay value already includes clock network latency in the external chip. Avoids double-counting.
Examples
# 2ns external logic, 500ps hold path — input constrained to CLK set_input_delay -max 2.0 -clock CLK_SYS [get_ports DATA_IN*] set_input_delay -min 0.5 -clock CLK_SYS [get_ports DATA_IN*] # DDR input: both edges set_input_delay -max 1.0 -clock CLK_DDR [get_ports DDR_DQ*] set_input_delay -max 1.0 -clock CLK_DDR -clock_fall -add_delay [get_ports DDR_DQ*]
⚠ Senior Note: Missing -min input delays leaves hold paths at input ports unconstrained. Always pair -max and -min for complete coverage. Ports without input_delay are analyzed with zero delay — dangerously optimistic.
set_output_delay I/O
Model external setup/hold requirement at an output port relative to a clock
set_output_delay [-max | -min] -clock <clock_name> [-clock_fall] [-rise | -fall] [-add_delay] [-network_latency_included] [-source_latency_included] delay_value port_list
Purpose
Specifies how much time the receiving chip (downstream) needs before the next clock edge. The output data must be stable this many ns before the downstream setup window. It's the downstream chip's setup requirement.
Effect on STA
Required arrival time = clock_period - output_delay (for max). The internal path to the output must complete within this window. A larger output_delay tightens internal logic. -min models downstream hold requirement.
Examples
# Downstream chip needs data 1.5ns before its clock edge set_output_delay -max 1.5 -clock CLK_SYS [get_ports DATA_OUT*] set_output_delay -min -0.2 -clock CLK_SYS [get_ports DATA_OUT*] # Negative min output_delay is valid — means data can arrive slightly after edge # (downstream has hold slack to spare)
💡 For I/O constraints: output_delay + internal_path_delay ≤ clock_period - setup_margin. The -min value can be negative if the downstream chip has a negative hold time requirement (typical in SDR SRAMs).

Timing Exceptions

5 commands
set_false_path Exception
Exclude a path from timing analysis — no setup or hold check required
set_false_path [-setup | -hold] [-rise | -fall] [-from <objects>] [-to <objects>] [-through <objects>] [-rise_from | -fall_from | -rise_to | -fall_to | -rise_through | -fall_through] [-comment <string>]
Purpose
Marks a timing path as non-functional or irrelevant for analysis. The path exists logically but never needs to meet timing because it's either asynchronous, a test path, a static config signal, or functionally impossible.
Effect on STA
The path is completely removed from setup/hold checks. No slack is computed, no violation is flagged. Overuse can hide real issues — always document with -comment.
FlagDescription
-fromStart point of the false path — clock, register, port, or pin.
-toEnd point of the false path.
-throughA waypoint — the path must pass through this pin or net. Used to narrow scope so only specific paths through a shared logic block are false.
-setupOnly exclude from setup (max delay) analysis. Hold still checked.
-holdOnly exclude from hold (min delay) analysis.
-rise / -fallOnly apply to rising or falling transitions on the path endpoint.
Examples
# Async CDC: no timing between clock domains set_false_path -from [get_clocks CLK_A] -to [get_clocks CLK_B] # Async reset pin (always arrives before design is active) set_false_path -from [get_ports ASYNC_RST_N] # Scan path — only false during scan mode (use case analysis too) set_false_path -through [get_pins */SE] # False only for setup check on a specific crossing set_false_path -setup -from [get_registers CFG_REG*] -to [get_registers LOGIC_REG*]
⚠ Senior Note: Over-applying set_false_path is a major sign-off risk. Use -through to scope it precisely. If you write -from clk1 -to clk2 without synchronizers, make sure those paths truly never carry timing-critical data. Always add -comment "Reason: …" for traceability.
set_multicycle_path Exception
Allow a path more than one clock cycle to propagate (slow/pipelined paths)
set_multicycle_path [-setup | -hold] [-start | -end] [-from <objects>] [-to <objects>] [-through <objects>] [-rise_from | -fall_from | -rise_to | -fall_to] path_multiplier
Purpose
Relaxes setup check by allowing N clock cycles for data to propagate. Used for slow datapaths (multipliers, large adders, non-critical config paths) that intentionally take more than one cycle.
Effect on STA
Setup window = N × period. But hold check also shifts — you MUST also add a corresponding -hold multicycle path to prevent the tool from checking hold against an earlier launch edge. Forgetting -hold MCP causes phantom hold violations.
FlagDescription
path_multiplierInteger. 2 = 2 cycles allowed, 3 = 3 cycles, etc. Applied to setup check by default.
-setupShift the capture edge forward by (N-1) clock cycles. Setup window becomes N×period.
-holdShift hold check reference. MUST pair with -setup MCP. For a 2-cycle MCP: set_multicycle_path -hold 1 to move hold check back to 1 cycle before the new capture edge.
-startThe multiplier is counted from the launch clock (source). Used when clocks differ and you want to move by launch cycles.
-endThe multiplier is counted from the capture clock (destination). Default for same-clock MCPs. Important for CDC paths.
Examples
# 32-bit multiplier takes 2 clock cycles — standard MCP pair set_multicycle_path -setup 2 -from [get_cells MULT_REG_IN*] -to [get_cells MULT_REG_OUT*] set_multicycle_path -hold 1 -from [get_cells MULT_REG_IN*] -to [get_cells MULT_REG_OUT*] # Config register read — 4 cycles allowed set_multicycle_path -setup 4 -from [get_registers CFG_*] set_multicycle_path -hold 3 -from [get_registers CFG_*]
⚠ Senior Note: The -hold MCP value = (setup_multiplier - 1) for same-domain same-frequency paths. For cross-domain paths with -start/-end, the math changes — draw a waveform diagram before applying. A wrong -hold value is a real hold violation risk in silicon.
set_max_delay Exception
Override maximum (setup) delay requirement on a specific path
set_max_delay [-datapath_only] [-from <objects>] [-to <objects>] [-through <objects>] [-rise | -fall] [-ignore_clock_latency] delay_value
Purpose
Sets an explicit maximum path delay constraint that overrides the clock-derived requirement. Used for paths that cross asynchronous domains but still need a bounded delay (e.g., synchronizer input must settle within N ns).
Effect on STA
The required arrival time at the endpoint is set to delay_value. Without -datapath_only, clock uncertainty and skew are still applied. With -datapath_only, it's a pure combinational delay constraint — no clock effects considered.
FlagDescription
-datapath_onlyIgnore all clock effects (latency, uncertainty, skew). The constraint is purely: actual_path_delay ≤ value. Used for CDC synchronizer constraints. Note: -datapath_only disables hold check on that path.
-ignore_clock_latencyExclude clock latency from the timing calculation but keep uncertainty effects. Less strict than -datapath_only.
Examples
# Path must complete within 3.5ns regardless of clock set_max_delay 3.5 -from [get_cells SYNC_FF1/Q] -to [get_cells SYNC_FF2/D] # Pure datapath CDC: settle within 4ns (pure combinational limit) set_max_delay -datapath_only 4.0 -from [get_clocks CLK_A] -to [get_pins U_SYNC/D]
⚠ Senior Note: -datapath_only suppresses hold checks. If you use it for a CDC path, ensure you have a separate set_min_delay or the synchronizer is robust. In Synopsys PrimeTime, -datapath_only is the preferred way to model metastability windows on CDC paths.
set_min_delay Exception
Override minimum (hold) delay requirement on a specific path
set_min_delay [-from <objects>] [-to <objects>] [-through <objects>] [-rise | -fall] delay_value
Purpose
Sets a minimum delay requirement (hold constraint) on a path that overrides the clock-derived hold check. The actual path delay must be at least delay_value ns.
Effect on STA
Useful for paths where you need to ensure a minimum propagation delay (e.g., for signal integrity, synchronizer metastability window, or to model minimum hold at destination). Less commonly used than set_max_delay.
Examples
# Ensure at least 0.5ns minimum hold on output port set_min_delay 0.5 -to [get_ports DATA_OUT]
set_disable_timing Exception
Disable a specific timing arc within a cell or between pins
set_disable_timing [-from <pin_name>] [-to <pin_name>] cell_or_net_list
Purpose
Removes a specific timing arc from a cell's timing model for analysis. Used when a particular input-to-output arc in a cell is not functionally active (e.g., a latch enable arc when latch is always transparent, or a mux select arc).
Effect on STA
The arc is completely invisible to the STA engine. No delay contribution from that arc. More surgical than set_false_path — affects the cell model, not just a path.
Examples
# Disable the enable-to-output arc of a clock gate (when not gating) set_disable_timing -from EN -to ENCLK [get_cells CG_CELL*] # Disable a feedback arc to break a combinational loop set_disable_timing -from Q -to D [get_cells LATCH_INST]
⚠ Senior Note: Be careful disabling arcs in latch-based designs — you might disable a real timing path. This should be used only when the arc is provably non-functional by design. Document every use.

Environment & Operating Conditions

4 commands
set_operating_conditions Environment
Set the PVT (process/voltage/temperature) corner for timing analysis
set_operating_conditions [-library <lib_name>] [-min <condition>] [-max <condition>] [-min_library <lib>] [-max_library <lib>] [-analysis_type on_chip_variation | bc_wc | single] [condition_name]
Purpose
Selects the PVT operating condition from the Liberty library (.lib) to use for cell delay calculations. Each operating condition specifies process corner, supply voltage, and temperature — which affects delay values.
Effect on STA
All cell and wire delays are scaled according to the selected condition. "slow" corners increase delays (pessimistic setup). "fast" corners reduce delays (pessimistic hold). -analysis_type on_chip_variation enables OCV derating.
FlagDescription
-minOperating condition for hold (min delay) analysis — usually the fastest corner (bc = best case).
-maxOperating condition for setup (max delay) analysis — usually the slowest corner (wc = worst case).
-analysis_type on_chip_variationEnables OCV mode where launch and capture paths can be analyzed at different PVT points.
-analysis_type bc_wcBest-case/worst-case: min analysis uses bc library, max analysis uses wc library. Traditional approach.
Examples
# Typical corner set_operating_conditions tt_0p8v_25c # Worst-case setup, best-case hold (bc_wc mode) set_operating_conditions -max ss_0p72v_125c -min ff_0p88v_m40c # OCV analysis (single library, derates applied separately) set_operating_conditions -analysis_type on_chip_variation ss_0p72v_125c
set_wire_load_model Environment
Assign a statistical wire load model to estimate routing parasitics pre-layout
set_wire_load_model -name <model_name> [-library <lib>] [-min | -max] [object_list]
Purpose
Defines wire resistance and capacitance estimates for nets as a function of fanout. The model is statistical — defined in the .lib file for a given die area/process. Used exclusively pre-layout.
Effect on STA
Adds estimated RC wire delay to all nets. Without this, only cell delays exist — nets have zero delay, which is overly optimistic for pre-layout closure checks. Post-layout replaces this with actual SDF/SPEF parasitics.
Examples
# Apply 10K-gate area wire load model set_wire_load_model -name WireLoad_10K -library my_process_lib # Different models for setup and hold set_wire_load_model -name WireLoad_10K_wc -max set_wire_load_model -name WireLoad_10K_bc -min
💡 Modern advanced node flows (7nm, 5nm) skip wire load models entirely — parasitic extraction from DEF is available early. Wire load models are primarily a 28nm and above technique.
set_wire_load_mode Environment
Control how wire load models are applied across hierarchical boundaries
set_wire_load_mode top | enclosed | segmented
ModeDescription
topAll nets in all blocks use the top-level wire load model. Simple but overly pessimistic for sub-blocks.
enclosedEach net uses the wire load model of its enclosing (lowest hierarchical) block. More accurate for hierarchical designs.
segmentedFor nets crossing hierarchy, each segment uses the wire load model of the module it traverses. Most accurate.
Examples
set_wire_load_mode enclosed # Most common for hierarchical designs
set_wire_load_selection_group Environment
Automatically select wire load model based on fanout count
set_wire_load_selection_group [-library <lib>] [-min | -max] group_name [object_list]
Purpose
Points to a wire_load_selection table in the Liberty library that maps net fanout ranges to specific wire load models. The tool picks the right model automatically per net.
Effect on STA
Nets with high fanout get larger wire load models; low-fanout nets get smaller ones. More accurate than a fixed single model.

Design Rule Constraints

5 commands
set_max_transition DRC
Set maximum allowable slew (transition time) on nets or ports
set_max_transition [-clock_path | -data_path | -rise | -fall] [-cell_check] transition_value object_list
Purpose
Limits how slow a signal edge can be (rise/fall time). Slow transitions cause increased short-circuit power, degraded noise margins, increased delay, and can even cause logic errors in some cell types.
Effect on STA
The tool reports DRC violations when computed slew exceeds this value. In synthesis, it drives buffer insertion to fix slew degradation. This constraint does NOT change timing — it flags violations.
FlagDescription
-clock_pathApply only to clock paths. Typical: 150–200ps for clock, stricter than data.
-data_pathApply only to data paths.
-cell_checkCheck against cell-level slew limits from the Liberty library, not just the SDC value. Use both together for comprehensive checking.
Examples
# 200ps max slew on entire design set_max_transition 0.2 [current_design] # Stricter 100ps on clock paths set_max_transition -clock_path 0.1 [current_design]
set_max_fanout DRC
Limit the maximum fanout (load count) on a net or design
set_max_fanout fanout_value object_list
Purpose
Constrains how many cells can be driven by a single net. High fanout causes degraded slew (slower transitions), increased delay, and sensitivity to noise. The tool inserts buffers to satisfy this.
Effect on STA
A DRC violation is flagged when fanout exceeds the limit. Synthesis/optimization inserts buffers or resizes drivers to fix. The indirect effect is improved slew and delay on high-fanout nets.
Examples
set_max_fanout 20 [current_design] set_max_fanout 8 [get_ports CTRL_*] # Tighter on control signals
set_max_capacitance DRC
Limit maximum capacitive load on a net or port
set_max_capacitance [-clock_path | -data_path] capacitance_value object_list
Purpose
Restricts the total lumped capacitance (pF) on a net. High capacitance causes slow transitions, increased dynamic power, and delay. Often derived from the Liberty cell's max_capacitance attribute.
Effect on STA
Reports DRC violation when total net cap exceeds limit. Synthesis adds buffers or uses stronger drive cells to fix. Affects transition times and consequently cell delays.
Examples
# Max 100fF capacitance on all nets set_max_capacitance 0.1 [current_design] set_max_capacitance -clock_path 0.05 [current_design]
set_min_capacitance DRC
Set minimum capacitive load requirement (ensures driver stability)
set_min_capacitance capacitance_value object_list
Purpose
Some drivers (particularly analog-influenced or mixed-signal) require a minimum load capacitance for stability (e.g., oscillator buffers, LDO outputs). This models that requirement.
Effect on STA
Rarely used in pure digital STA. The tool flags nets with total cap below the minimum. Synthesis may add dummy loads (tie cells) to meet this — use cautiously.
set_max_area DRC
Constrain the maximum combinational area of the design
set_max_area area_value
Purpose
Specifies the maximum total gate area (in library units, typically µm²) the synthesized design can occupy. Guides the synthesis tool to use smaller cells and minimize logic duplication.
Effect on STA
Primarily a synthesis constraint. In STA, it affects which cells are chosen during ECO. Setting to 0 means "minimize area as much as possible while meeting timing."
Examples
set_max_area 0 # Minimize area (no hard limit) set_max_area 500000 # Hard limit: 500,000 µm²

Port & Cell Modeling

6 commands
set_load Port/Cell
Assign a capacitive load value to output ports or nets
set_load [-min | -max] [-pin_load | -wire_load] [-subtract_pin_load] capacitance_value object_list
Purpose
Models the external capacitive load driven by an output port (PCB trace, next-stage input capacitance, ESD structure). Without this, output port has only internal cell capacitances — overly optimistic for output timing.
Effect on STA
Added load increases output transition time, which increases delay of the driving cell. This propagates backward and can tighten setup slack on the path leading to that output. Critical for I/O timing accuracy.
Examples
# 5pF board trace capacitance on all output ports set_load 5.0 [all_outputs] # Different load for setup vs hold set_load -max 6.0 [get_ports BUS_OUT*] set_load -min 4.0 [get_ports BUS_OUT*]
set_drive Port/Cell
Model external driver resistance at input ports
set_drive [-rise | -fall] [-min | -max] drive_resistance input_port_list
Purpose
Simulates the resistance of the external driver at an input port. A non-zero drive resistance creates an RC delay at the input, affecting the slew of the first cell in the path.
Effect on STA
The input port slew is derived from drive_resistance × input_pin_capacitance. This slew feeds into the first cell's delay lookup table. Preferred over set_drive: use set_driving_cell for more accuracy.
Examples
# 50 ohm source resistance (typical PCB) set_drive 50 [all_inputs] set_drive 0 [all_clocks] # Ideal drive for clocks
set_driving_cell Port/Cell
Model external driver at input port using a real library cell (preferred over set_drive)
set_driving_cell -lib_cell <cell_name> [-library <lib_name>] [-pin <pin_name>] [-from_pin <input_pin>] [-rise | -fall] [-min | -max] [-no_design_rule] [-dont_scale] [-input_transition_rise <t>] [-input_transition_fall <t>] input_port_list
Purpose
Specifies the exact library cell (e.g., BUFX8, INV4) that drives a given input port. The tool looks up this cell's drive strength from the Liberty model to compute accurate input slew, rather than using a simple resistive approximation.
Effect on STA
Most accurate method for modeling I/O port drivers. The cell's output resistance tables determine the slew delivered to the first internal cell. Directly impacts delay accuracy at input paths.
Examples
# Use BUFX8 from std cell lib to drive all inputs set_driving_cell -lib_cell BUFX8 -library sc_lib [all_inputs] # Use specific output pin of a driving cell set_driving_cell -lib_cell MX2X4 -pin Y [get_ports SEL_*]
💡 set_driving_cell is preferred over set_drive because it uses actual NLDM/CCS driver models from the .lib, giving accurate slew computation. Always use set_driving_cell for sign-off quality.
set_input_transition Port/Cell
Directly assign a slew value to input ports (simpler alternative to set_driving_cell)

Purpose
Assigns a fixed slew time to the input port signal. Simpler than set_driving_cell but less accurate since it doesn't model how load-dependent the external driver is.
Effect on STA
The specified transition is used directly in timing arc calculations for the first cell receiving this input. Use when the external driver's exact characteristics are unknown.
Examples

set_fanout_load Port/Cell

set_fanout_load fanout_value port_list
Purpose
Assigns a numerical fanout load to an output port. This is counted against the set_max_fanout limit on the net driving this port. Models the downstream fanout the port contributes.
Effect on STA
Affects fanout DRC checking. If the external fanout load pushes the total over the limit, the tool reports a violation and may insert buffers.
set_port_fanout_number Port/Cell
Set estimated number of external fanout connections for an output port
set_port_fanout_number number port_list
Purpose
Provides the STA tool with the expected number of loads the output port drives externally. This is used to select an appropriate wire load model for the port.
Effect on STA
The wire load model lookup uses this number to size the parasitic model. Higher fanout → larger RC wire model → more pessimistic output delay.

group_path Path Groups
Create a named path group for organized reporting and targeted optimization
group_path -name <group_name> [-from <objects>] [-to <objects>] [-through <objects>] [-weight <value>] [-critical_range <range>] [-default]
Purpose
Groups timing paths into named categories (e.g., reg2reg, in2reg, reg2out, in2out, feedthrough). This controls both which paths are reported together and how the optimizer prioritizes fixing violations.
Effect on STA
The synthesis/optimization tool uses path group weights to prioritize. Higher weight = higher fixing priority. Reporting tools (report_timing) show paths per group. Default groups are created per clock automatically.
FlagDescription
-nameGroup name visible in reports and used by the optimizer to identify this category.
-weightRelative priority: higher weight → optimizer focuses more effort. Default = 1.0.
-critical_rangePaths within this range (ns) of the worst negative slack are treated as critical and heavily optimized.
-defaultAssigns all remaining unmatched paths to this group.
Examples

set_timing_derate Path Groups
Apply OCV/AOCV/POCV derating factors to cell and net delays
set_timing_derate [-cell_delay | -cell_check | -net_delay] [-data | -clock] [-early | -late] [-rise | -fall] [-from <objects>] [-to <objects>] [-through <objects>] derate_factor
Purpose
Models On-Chip Variation (OCV): delays along the launch path and capture path are derated differently to account for spatial variation across the die. A factor > 1.0 makes delays pessimistic (late), < 1.0 makes them optimistic (early).
Effect on STA
For setup: launch path is derated late (slowed) and capture path is derated early (sped up), tightening setup margin. For hold: launch early, capture late. This is the industry-standard OCV methodology for sign-off.
FlagDescription
-cell_delayApply to cell propagation delays (combinational and sequential arcs).
-cell_checkApply to setup/hold timing checks (su/hold windows of registers).
-net_delayApply to net/interconnect delays (RC parasitics).
-data / -clockApply only to data paths or clock paths respectively.
-early / -late-late derates toward pessimistic (delays increase). -early derates toward optimistic (delays decrease).
Examples
# Standard flat OCV: 5% pessimism on late, 5% optimism on early set_timing_derate -late -cell_delay 1.05 # launch path 5% slower set_timing_derate -early -cell_delay 0.95 # capture path 5% faster # Clock path gets more aggressive derating than data set_timing_derate -clock -late -cell_delay 1.08 set_timing_derate -clock -early -cell_delay 0.92
⚠ Senior Note: AOCV (Advanced OCV) and POCV (Parametric OCV) replace flat derating at advanced nodes. AOCV uses depth-based derate tables; POCV uses sigma-based statistical distributions. set_timing_derate is still needed for AOCV/POCV but the values come from library tables, not fixed percentages.
set_clock_gating_check Path Groups
Define setup/hold requirements for clock gating enable signals
set_clock_gating_check [-setup <time>] [-hold <time>] [-rise | -fall] [-high | -low] [object_list]
Purpose
For integrated clock gating (ICG) cells, the enable signal must be stable before the clock gating window opens (setup) and remain stable after it closes (hold). This command defines those requirements when they differ from what the Liberty model specifies.
Effect on STA
Adds a timing check between the enable pin and the gated clock output. Violations mean glitchy clocks can propagate, causing functional failures. The tool may flag EN-to-GCLK paths as new timing paths requiring fixing.
Examples
# 200ps setup, 100ps hold for all ICG enable pins set_clock_gating_check -setup 0.2 -hold 0.1 # Override for specific ICG cells in power domain A set_clock_gating_check -setup 0.3 -hold 0.15 [get_cells PD_A/ICG*]
set_data_check Path Groups
Define a timing relationship (setup/hold) between two data signals
set_data_check [-setup <time>] [-hold <time>] [-rise_from | -fall_from | -from] [-rise_to | -fall_to | -to] [-clock <clock>]
Purpose
Creates a non-clock timing check between two data pins. Used when one data signal acts as a quasi-clock or enable for another — for example, a write-enable arriving before an address strobe. Not a conventional STA check, but powerful for special cases.
Effect on STA
The STA tool inserts a timing check arc: the -to pin must arrive at least -setup time before, and at most -hold time after, the -from pin's transition. Violations indicate temporal ordering issues between control signals.
Examples
